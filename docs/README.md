# Visual With
*포폴용 리드미*

## 프로젝트 소개
Visual With는 팀 협업에 필요한 핵심 기능들을 기능 단위 구조로 설계해 제공하는 웹 기반 협업 툴입니다.

장기 팀 프로젝트를 전제로, 기능 확장과 유지보수를 고려한 폴더 구조와 상태 관리 설계를 중심으로 개발했습니다.
복잡한 기능 나열보다는, 사용자가 기능의 흐름을 쉽게 이해할 수 있는 인터페이스를 목표로 했습니다.


## 시작 방법
```
npm install
npm run dev
```
http://localhost:5173


## 주요 기능

### 1. 사용자 인증 및 접근 흐름
- 로그인 및 회원가입  
  → 사용자 인증을 기반으로 개인/팀 단위 기능 접근 제어

### 2. 팀 기반 협업 기능
- 팀 초대 및 팀 전환  
  → 하나의 계정으로 여러 팀 컨텍스트를 전환하며 작업 가능

### 3. 팀 내부 협업 기능
- 채팅, 일정, 파일, 할 일 관리  
  → 팀 단위 커뮤니케이션과 업무 관리를 하나의 인터페이스에서 제공

### 4. 개인 간 소통 기능
- 친구 간 채팅  
  → 팀과 분리된 1:1 커뮤니케이션 경로 제공

- 기능 구현 현황 및 로드맵: [features.md](features.md)

## 기술 스택 및 도구

### 핵심 기술 스택

### 1. React & TypeScript
- 역할 분리: 협업 툴 기반 프로젝트에서 컴포넌트 단위 역할 분리 및 개발 생산성 향상
- 안정성: TypeScript를 활용해 컴포넌트 간 데이터 전달 시 발생 가능한 오류 사전 방지

### 2. Context API
- 공통 상태 관리: 로그인 정보, 현재 팀(teamId) 등 여러 화면에서 쓰이는 데이터 관리
- 전역 상태 기준 정립: 단순 전역 상태 사용이 아닌, “어디까지를 전역으로 가져가야 하는가”에 대한 기준을 세우는 경험
- 성능 최적화 고민: 불필요한 리렌더링 문제를 경험하며 전역 상태 범위와 책임 분리에 대해 고민

### 3. CSS Modules
- 스타일 분리: 컴포넌트 단위 스타일 적용으로 클래스명 충돌 방지 및 스타일링 책임 분리
- 스타일 기준 재정립: 컴포넌트를 무조건 잘게 쪼개기보다, 변경 빈도와 스타일 응집도를 기준으로 분리 단위를 재조정함

### 4. Storybook
- UI 개발 효율: 앱 전반에서 재사용되는 버튼, 입력창 등 공통 컴포넌트를 독립적으로 확인하고 수정
- 의도치 않은 스타일 변경 방지: 기능 개발 중 스타일 변경이 다른 화면에 미치는 영향을 줄이기 위해 도입

## 고민했던 점 & 해결 과정

### 1. JavaScript에서 TypeScript로의 전환을 통한 안정성 확보

* **문제**: 초기 JavaScript 기반 구현 시, 프로젝트가 장기화됨에 따라 유지보수성과 협업 측면에서 데이터 구조 파악 및 에러 방지에 한계를 느낌.
* **해결**: 학습 비용을 감수하고 TypeScript로 전환하여 컴포넌트 설계와 상태 관리 시 활용되는 데이터 구조를 인터페이스로 명확히 정의함.
* **결과**: 기능 확장 과정에서 발생하던 데이터 사용 오류를 사전에 방지하고, 데이터 흐름을 명확하게 파악할 수 있는 환경을 구축함.

### 2. 채팅 데이터 구조의 객체화 및 확장성 개선

* **문제**: 초기 채팅 데이터를 단순 문자열(string)로 구성하여 메시지 전송 시간 표시 등 추가 요구사항 대응 시 로직이 복잡해지는 문제를 인식함.
* **해결**: 메시지 내용뿐만 아니라 시간, 날짜, 보낸 사람 등 다양한 정보가 포함될 수 있도록 채팅 데이터를 객체 배열 형태로 재설계함.
* **결과**: 데이터 출력부에서 정보를 다루는 복잡도를 낮췄으며, 향후 추가될 정보에 유연하게 대응할 수 있는 데이터 구조를 확보함.

### 3. 유지보수 효율을 위한 기능 단위(Features) 구조 도입

* **문제**: 프로젝트 규모가 커짐에 따라 공통 `components` 폴더 내 컴포넌트 간 의존성이 높아지고 스타일 관리가 어려워짐.
* **해결**: 기능 단위로 페이지와 컴포넌트를 관리하는 `features` 기반 구조를 도입하고, 재사용성과 스타일 관리 비용을 기준으로 컴포넌트 분리 기준을 재정의함.
* **결과**: 새로운 기능 추가 시 기존 코드에 미치는 영향을 최소화하고, 특정 기능에 종속된 컴포넌트의 책임을 명확히 분리함.

### 4. 레이아웃 분리를 통한 스타일 의존도 개선

* **문제**: 메인 레이아웃과 개별 기능 페이지 간의 스타일 의존도가 높아져 레이아웃 수정 시 여러 페이지를 개별 수정해야 하는 번거로움 발생.
* **해결**: `MainBody` 중간 레이어를 생성하여 공통 스타일을 제어하고, 페이지 구조 담당 컴포넌트(`layouts`)와 일반 UI 컴포넌트를 폴더 수준에서 분리함.
* **결과**: 레이아웃 수정 시 하위 기능 페이지를 수정하지 않아도 되는 독립적인 구조를 확보함.

### 5. 렌더링 최적화 및 코드 재사용성 강화

* **문제**: 채팅 입력 시 입력값 변경만으로 채팅 뷰 전체가 리렌더링되는 성능 저하 확인 및 팀/친구 채팅 간 로직 중복 발생.
* **해결**: `React.memo`를 적용하여 메시지 리스트 컴포넌트를 분리하고, 중복되는 채팅 로직을 공통 뷰 컴포넌트로 통합하여 데이터 주입 방식으로 개선함.
* **결과**: 불필요한 리렌더링을 방지하여 성능을 개선하고, 동일한 UI/로직을 가진 기능의 코드 중복을 제거함.

### 6. 환경 간 불일치 방지를 위한 명명 규칙 통일 및 Git 설정 해결

* **문제**: 폴더명 대소문자 혼용으로 인해 Git에서 변경 사항을 인식하지 못하거나, 로컬과 실행 환경의 결과가 일치하지 않는 문제 발생.
* **해결**: 전체 폴더명을 소문자로 통일하고, Git의 대소문자 구분 설정(`core.ignorecase`)에 대해 학습하여 문제를 해결함.
* **결과**: 일관성 있는 파일 명명 규칙의 중요성을 체감하고, 개발 환경 설정이 프로젝트 안정성에 미치는 영향을 이해함.

## 아쉬운 점 & 개선 계획
- 특정 컴포넌트가 무거워지며 성능 저하 경험
  → React Developer Tools로 원인을 분석하고, 렌더링 최적화 및 컴포넌트 책임 재분리 고려
- Context API가 복잡해짐에 따라 불필요한 리렌더링 발생
  → 추후 Zustand 도입으로 개별 상태 관리 예정
- 개인 채팅의 오버플로우 현상, 창 크기 변경 시 UI 깨지는 문제
  → 어떤 컴포넌트의 스타일로 원인이 발생하는지 확인하여 해결 예정
- api 파일이 커지면 각 기능별로 파일을 분리하여 관리할 예정
